## <a name="intern"></a>상반기 네이버 인턴을 돌아보며, 임문수님

### 인턴에 들어오기 까지

- 상,하반기 인턴채용
- 인턴 수시채용
- 프로그램 연계채용 (부스트캠프)

#### 부스트캠프 지원에 필요한것

- 자바스크립트는 필수, 타입스크립트도 하면좋다.
- Git (branch, checkout, commit, push, rebase, reset) 정도는 알아보자.
- 토이 프로젝트
  - 혼자 해야한다면 todo같은것. 팀은 서비스 클론으로
  - 개발에만 집중이 아니라 설계, 일정산출, 기능개발 같이 프로세스를 지키면서 진행하기
  - 기본적인 알고리즘 테스트(코딜리티로 진행), 인성, 기술 면접으로 진행한다.
  - CS 질문 : [Technical Interview Guidelines for Beginners](https://github.com/JaeYeopHan/Interview_Question_for_Beginner)

#### 기술면접?

- 키워드의 개념도 중요하지만 더 심도있게 공부하면 좋다
- 왜 이런 기술이 도입되었는가, 어떻게 활용할 수 있는가?, 다른 기술들과 어떻게 다르고 어떤 장단점이 있는가?

<br>

### 인턴동안 무엇을 했나요

- 1주차 (적응기간)
  - 대여기기에 개발준비
  - 간단한 과제 (자바스크립트, 웹 기반 지식)
  - 프로젝트에 필요한 사전 강의나 문서
  - 개발에 필요한 팁
- 2~7주차 프로젝트 개발기간
  - 일정 산출, 설계
  - 프로젝트 개발
  - 프로젝트 회의 및 주간 회의
- 8주차 최정 발표기간
  - 프로젝트 시연 준비
  - 발표자료 준비 및 발표

<br>

### 좋았던 시도들

1. 테스트 코드
2. 꼼꼼한 코드 리뷰

### 아쉬웠던 시도들

- 프로젝트 및 개인 일정 관리 : 생각보다 개발에 집중할 수있는 시간은 적었다.
  - 근무시간이 8시간이니까 6시간 투자할줄 알았는데 그렇지 않으셨다고 함
  - 일정이 밀린다면 우선적으로 개발할 것들을 개발하고 다음 단계로 넘어가자
- 커뮤니케이션
  - 애매한 용어 사용을 자제하자
  - 글로 의견을 전달하게 된다면 너무 딱딱한 표현은 피하자
  - 내가 말하고 싶은 내용보다는 상대방의 궁금해하는 내용을 고민하고 이해해기 쉽게 전달하자

- 직접 질문하기 어렵다면 고민을 잘 공유해보자

## <a name="open"></a>네이버에서 오픈소스 개발을 통해 성장하는 방법, 최연규님

[https://naver.github.io/egjs/](https://naver.github.io/egjs/)라는 오픈소스를 운영하신다고 한다.

### 오픈소스란?

오픈소스 소프트웨어는 저작권 소유자가 사용자에게 소프트웨어를 누구에게나 어떤 목적으로든 사용, 연구, 변경 및 배포할 수 있는 권한을 부여하는, 라이센스에 따라 소스 코드가 공개되는 일종의 컴퓨터 소프트웨어

ex) Git, Nodejs, npm, VSCode 등

<br>

### 왜 오픈 소스를 사용할까??

잘 만들어진 외부 오픈 소스를 사용함으로써 개발 비용을 절감할 수 있다.

<br>

### 왜 오픈소스를 만들까 왜 오픈소스로 만들까

- 우리(나)의 지식 공유를 할 수 있다
- 우리가 하는 일을 알릴 수 있다
- 커뮤니티를 형성하여 사용자들의 참여를 기대할 수 있다
  - 버그 제보 및 보안 개선, 새로운 기능 제안, 이를통해 더 나은 제품을 만들 수 있음
- 학습할 수 있는 기회가 많이 생긴다
  - 가이드가 있어도 사용자들마다 모두 똑같은 방법으로 사용하지 않는다
  - 그렇기 때문에 다양한 사용법에 따른 다양한 상황에서 다양한 버그제보를 받을 수 있다

<br>

### 오픈소스 참여 - 이유?

1. 사용했는데 생각했던 대로 잘 동작하지 않는다
2. 사용했는데 버그 발견
3. 사용했는데 기능 부족
4. 눈에 보이는 오타
5. 번역
6. 기여하고 싶을 정도로 흥미로운 프로젝트
7. 기타

> 최연규님의 발표는 나에게는 좀 어려웠다. 그래도 평소에 오픈소스라는 단어는 많이 들어봤지만, 그래서 오픈소스라는 것이 무엇인지, 어떻게 만들어지고 운영되는가에 대해서는 잘 몰랐는데 그런 것들을 알 수 있었던 좋은 경험이었다.

<br>

## <a name="youcan"></a>야 너두 할 수 있어. 비전공자, COBOL 개발자를 거쳐 네이버에서 FE 개발하게 된 이야기, 차성원님

> 차성원님은 아빠는 개발자라는 블로그를 운영하고 계시고, 경영학을 전공한 비전공자셨다.

- 평생 공부해야하는 것이 개발자. 끊임 없이 기술들이 나옴
- 진짜 개발이 즐거운 건지 스스로에게 질문해보자

### 다똑같은 개발자가 아니다

게임도 rpg, 전략 시뮬 등등 다양하듯 개발도 두루 경험해보자

#### 실력이 부족한지 흥미가 없는건지 모르겠어요

남과 비교하지말고 작은것부터 만들어보자. 그때 뿌듯함, 성취감이 쌓이면 실력이 된다고 생각한다. 본인이 재미있는 분야를 찾자.

<br>

### 취업준비

채용할때 주로 보는 관점이 다르지만, 신입은 **기본기**를 많이 본다.

#### 그래서 기본기가 대체 무엇인가요?

> 네이버 윤종호님

기술 생태계는 변화가 빠르기 때문에, 의욕만 앞서 귀납적으로 모두 다 경험해보겠다는 태도는 그다지 효율적인 접근 방법이 아니다. 이럴수록 기본기에 충실해야지만, 새로운 기술 흐름에 압도되지 않고, 스스로의 방향성과 페이스로 앞으로 나아갈 수 있다.

FE, BE, 데이터와 관련해 하루가 멀다하고 새로운 구현체들이 온갖 미사여구로 무장하여, 공부하지 않으면 마치 도태될 것 같은 느낌을 줄 텐데, 이들 기저의 **구현 철학과 원리를 이해**하고, 우리가 사용하려는 **응용의 특성에 맞게 잘 활용할 수 있는 연역적인 혜안을 갖추는 것**이 바로 기술 기본기이다.

더불어 이들 기술 기본기는 명멸하는 구현체들에 비해, 그렇게 빠른 속도로 변하지 않고 축적되기 때문에, 효율성 차원에서라도 먼저 우선순위를 가지고 투자할 가치가 있다.

<br>

### 기억할 것들

- 변화의 흐름을 읽고, 적절한 기술을 선택하는게 중요
- 어떤게 핫해서 쓰는게아니라, 어떤걸 했어요가 아니라, 어떻게가 중요하다
- 프로젝트를 끝낼때마다 아쉬웠던점을 기록하면 좋다

<br>

> 차성원님의 발표는 개발 공부를 하는 방향에 있어 도움이 많이 되었다. 빠르게 변화하는 개발 생태계에서 변화에 따라 새로운 기술만 쫓는 것이 아닌, 어떤 기술을 선택하더라도 왜, 어떻게 사용하는 것인가를 알고 쓰는 것이 중요하다고 느꼈다.

## <a name="review"></a>Code Review is Horse(코드리뷰는 말이야)(feat.Latte), 오두경님

### 코드리뷰란?

여러 사람들이 코드를 읽고 검증하여 소프트웨어 품질을 노피는것

- 리뷰어 - 작성한 코드를 보고 의견을 주는 사람
- 리뷰이 - 코드를 작성하고 다른사람에게 의견을 받는사람

#### 네이버 개발자들은 코드리뷰를 어느정도 하고 있나요?

- 약 95% 개발자들이 코드 리뷰를 하고 있다.

<br>

### 코드 리뷰 단계

### 0단계 : 한 눈에 오류가 보이는가

- 우리가 제일 많이하고 쉽게 할 수 있는 리뷰
  - 공백
  - 오타
  - 줄바꿈
  - 변수명
  - 코드 컨벤션

### 코드 리뷰를 도와주는 도구들

- 자동화
  - prettier : 에디터에 저장 시 포매팅 설정
- VSCode
  - Code Spell Checker : 오타 방지 에디터 플러그인
- Lint
  - eslint. JSLint/JSHint
- husky : 커밋, push등 git 훅을 설정할 수 있는 라이브러리
  - fix lint, prettier, 커밋메세지 컨벤션 적용

<br>

### 1딘계 : 기능 구현이 잘 되었는가

- 주어진 스펙이 구현이 잘 됐는가?
  - 스펙에 ~라는 부분이 있는데 이 부분이 기능에 누락된 것 같습니다
- 버그 찾기. 나아가 디버깅 하기
  - B페이지에서도 이 함수를 호출하고 있는데요
  - 현재 변경으로 인해 동작하지 않네요
  - 인자가 잘못 넘어오는 것 같습니다

#### 코드상 보이는 명백한 버그 찾기 vs 실행 시킨 후 버그 찾기

- 실행 시킨 후 버그가 더 찾기 어렵다

<br>

### 2딘계 : 확장성이 좋은 코드인가

- 반복을 줄일 수 있는가?

- 불필요한 코드가 있는가

### 코드 단위를 작게하라

- 리뷰하기 어려워짐. 리팩토링을 기약하고 머지될 수 있음
- 리뷰어가 diff를 열어보고, 읽기도 전에 지칠 수 있음

#### 작은 단위, 어떻게 하죠?

- 처음부터 하긴 당연히 어렵다
- 코드 단위는 개발자가 느끼는 주관적인 영역이기 때문에 작은 단위에 대해 객관적으로 정할 수 없기도 하다
- git diff 적극 활용하기
  - 불필요한 변경은 diff에 잡히지 않도록 함
  - 공백, 에디터 설정에 의한 많은 파일 변경
  - eslint comma-dangle 도 불필요한 diff와 관련된 룰임
- diff에 불필요한 내용이 들어가지 않도록 커밋 나누기
- 로직과 단순 파일이름 변경 삭제 커밋은 분리하기

<br>

### 코드를 예상 가능하게 하라

#### 예상 가능한 코드?

- 커밋 잘하기
  - 커밋 메시지로 코드 수정 이유를 명확하게 하기
- PR 잘쓰기
  - 프로젝트 관리방법은 팀마다 다를 수 있으니 코드베이스로 문서를 볼 수 있는 곳은 PR(Pull Request)
  - PR엔 코드를 이해하기 위한 모든 내용이 있으면 좋음
  - PR은 아무것도 모르는 사람이 본다고 생각하고 쓰기

<br>

### 남의 코드 이해하기

> 리뷰어의 노력에도 불구하고 코드가 이해가 가지 않는다면?

- 이해가 안가는 부분에 대한 명확한 근거 말하기
- 잘 이해가지 않는다면 오프라인 리뷰 요청하기

<br>

### 리뷰어도 좋고 리뷰이도 좋은 코드 리뷰하기

- 내코드 != 나
  - 많은 리뷰 상처받지 말자
  - 리뷰를 모두 반영하면 분명 성장해 있을것
- 핑거프린세스는 되지말자
  - 리뷰가 간단하더라도 리뷰어가 리뷰한 내용에 대한 근거는 찾아보자, 테스트해보자
    - 현재 구현부와 리뷰의 내용의 차이점을 찾아본다
    - 검색/테스트 해본다
    - 그래도 모르겠으면 리뷰를 반영하기 위한 노력을 어필하며 대답한다

#### 노력의 어필, 어떻게하죠?

- Case 검색 : 아마 이런 내용이 아니실까 하여 이렇게 찾아봤습니다. 하지만 그래도 리뷰 내용을 정확하게 이해하지 못했습니다. 이 부분을 리뷰 주신게 맞을까요?
- Case 테스트 : 보여주신 코드로 이렇게 테스트 해봤는데 (코드, 테스트한 브랜치가 있으면 더 좋음) 오히려 흐름이 복잡해져서 코드의 흐름이 잘 보이지 않게 되더라구요. 그래서 현재 지금 안을 유지하면 좋을 것 같은데, 리뷰 주신 내용을 제가 제대로 반영한게 맞을까요?

#### 피드백을 하자

- 리뷰 반영만 피드백이 아니다
- 좋은 리뷰는 이 부분을 알려줘서 좋았다! 라는 피드백을 주자
- 리뷰 반영시 시간이 오래 걸릴 경우는 리뷰어에게 알려주자

<br>

### 반영해야 할 리뷰 vs 했으면 하는 리뷰 구분하기

- 버그 발생 **할**
- 개인 스타일 차이 **하는**
- 기능상 문제는 없으나 코드 개선이 필요, 작업이 큰 경우
  - **할** : 코드 일관성, 유지 관리 측면
  - **하는** : 일정이 빠듯한 경우

<br>

> 오두경님의 발표는 사실 코드 리뷰를 경험할 기회가 거의 없다보니 공감하기가 어려웠다. 그러나 항상 글이나 말로만 얼핏 들어봤던 코드리뷰 라는 것에 대해 실제로 알게 되어 좋았고, 나중에 이렇게 코드 리뷰가 갖춰진 환경에서 참고한다면 정말 도움이 많이 될 것 같았다.
