## 테이블 설계

- **테이블은 관계형 데이터에서 데이터를 관리하고 저장하기 위한 그릇**
- **테이블은 공통적인 요소의 집합** => 관련성이 없는 것들이 이것저것 모이면 안된다.
- 각각의 행이 어떤 공통적인 특징을 가진 집합이어야 한다.
- 테이블 명은 반드시 복수형이나 집합명사로 표현해야 한다.

<br>

### 테이블 설계 규칙

- 사물과 사물의 집합은 계층이 다르다.
  - ex) 사과, 포도는 테이블의 행이 될수는 있어도 집합을 나타내는 테이블이 될순 없다.
  - 이런 테이블을 허가하면 과일을 추가할때마다 새로 테이블을 만들어야함
- 가장 상위 개념 집합으로 정리한다.
- 한 개 테이블의 내용에 중복행을 허락하지 않는다.
- 데이터가 정적이지 않고 **동적**임을 전제로 생각해야 한다.

<br>

#### 기본 키

- 고유하게 식별 가능한 정보인 **기본 키는 한 테이블에 하나만** 존재해야 한다.
- 기본키로 사용하는 열은 중복이 발생하지 않는 열을 할당해야 한다.
  - 이름은 기본키로 보통 부적절하다. 유동적이고 불안정함
- 기본키 열로 `NULL` 을 설정할 수 없다.
- 테이블을 표 형식으로 나타낼때는 일반적으로 기본키 열 명에 밑줄로 표시

| <u>회원ID (기본키)</u> | 이름 | 나이 | 성별 |
| :--------------------- | :--- | :--- | :--- |
| A1                     | 김OO | 22   | 남   |
| A2                     | 정OO | 25   | 남   |
| B1                     | 한OO | 29   | 여   |
| B2                     | 최OO | 33   | 남   |

#### 기본키 값이 바뀌면 안되는 이유

- 변경 후 값의 유일성을 보증할 수 없음
- 과거 데이터와의 결합이 어려움
