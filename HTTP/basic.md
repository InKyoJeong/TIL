## 웹 클라이언트와 서버

- 웹 서버는 인터넷의 데이터를 저장하고 HTTP클라이언트가 요청한 데이터를 제공함
- 클라이언트가 서버에게 HTTP요청을 보내면 서버는 요청된 데이터를 HTTP응답으로 돌려줌
- HTTP클라이언트와 HTTP서버는 월드 와이드 웹의 기본 요소

<br>

## 리소스

- 웹 서버는 웹 리소스를 관리하고 제공
- 가장 단순한 웹 리소스는 파일시스템의 정적파일
- 정적파일은 텍스트, HTML, MS워드, 어도비 아크로뱃 파일, JPG이미지, AVI동영상, 그외 모든 종류 파일을 포함
- 리소스는 반드시 정적파일일 필요는 없고 요청에따라 콘텐츠를 생산하는 프로그램이 될 수도 있음

<br>

## 미디어 타입

- 인터넷은 수천가지 데이터 타입을 다루기 때문에, HTTP는 웹에서 전송되는 객체 각각에 MIME타입이라는 데이터 포맷 라벨을 붙임
- 웹서버는 모든 HTTP 객체 데이터에 MIME 타입을 붙임
- 웹브라우저는 서버로부터 객체를 돌려받을때, 다룰수 있는 객체인지 MIME타입을 통해 확인
- 웹브라우저는 대부분 객체 타입 수백가지를 다룰 수 있음. ex) 이미지 파일을 보여주고, HTML파일 분석/포맷팅, 오디오파일 재생 등
- MIME 타입은 `/`로 구분된 주 타입(primary object type)과 부타입(specific subtype)으로 이루어진 문자열 라벨
  - HTML 텍스트 문서 : text/html 라벨이 붙음
  - plain ASCII 텍스트 문서 : text/plain 라벨이 붙음
  - JPEG 이미지 : image/jpeg 가 붙음
  - GIF 이미지 : image/gif 가됨
  - 애플 퀵타임 동영상 : video/quicktime 이 붙음

<br>

## URI

- 웹 서버 리소스는 각자 이름을 가져서, 클라이언트는 관심있는 리소스를 지목할 수 있음
- 서버 리소스 이름은 **통합 자원 식별자(Uniform Resource Identifier, URI)** 라고 불림
- URI는 인터넷의 우편물 주소 같은것. 정보 리소스를 고유하게 식별하고 위치를 지정할 수 있음
  - ingg의 웹서버에 있는 이미지 리소스에 대한 URI라면
  - http://www.ingg.com/datastructure/stack.gif
- URI에는 **URL, URN** 두가지가 있음

### URL

- 통합 자원 지시자(Uniform Resource Locator, URL)는 리소스 식별자의 가장 흔한 형태
- URL은 특정 서버의 한 리소스에 대한 구체적 위치를 서술
- 리소스가 정확히 어디에 있고 어떻게 접근할 수 있는지 알려줌
- URL은 세부분으로 이뤄진 표준 포맷을 따름
  - 첫번째 부분은 스킴(scheme)이라고 불림. 리소스에 접근하기 위해 사용되는 프로토콜을 서술. 보통 HTTP프로토콜(http://)
  - 두번째 부분은 서버의 인터넷 주소를 제공(ex: www.ingg.com)
  - 세번째 부분은 웹서버의 리소스(ex: /datastructure/stack.gif)
- 오늘날의 대부분 URI는 URL임

### URN

- 유니폼 리소스 이름(Uniform Resource Name, URN)
- 콘텐츠를 이루는 한 리소스에 대해, 그 리소스 위치에 영향받지않는 유일무이한 이름 역할
- 예를들어, 다음 URN은 인터넷 표준 문서 'RFC 2141'이 어디에 있건 그것을 지칭하기위해 사용가능
  - `urn:ietf:rfc:2141`

## 트랜잭션

- HTTP 트랜잭션은 요청명령(클라이언트에서 서버로 보냄)과 응답 결과(서버가 클라이언트에게 돌려줌)로 구성됨

### 메서드

- HTTP는 HTTP 메서드라 불리는 여러 종류 요청 명령을 지원함
- 모든 HTTP 요청 메시지는 한 개의 메서드를 가짐
- 메서드는 서버에게 어떤 동작이 취해져야하는지 말해줌
- 흔히 쓰이는 **HTTP 메서드**들
  - `GET` : 서버에서 클라이언트로 지정한 리소스를 보내라
  - `PUT` : 클라이언트에서 서버로 보낸 데이터를 지정한 이름의 리소스로 저장하라
  - `DELETE` : 지정한 리소스를 서버에서 삭제하라
  - `POST` : 클라이언트 데이터를 서버 게이트웨이 애플리케이션으로 보내라
  - `HEAD` : 지정한 리소스에 대한 응답에서, HTTP 헤더 부분만 보내라

### 상태 코드

- 모든 HTTP 응답 메세지는 상태 코드와 함께 반환됨
- 상태 코드는 클라이언트에게 요청이 성공했는지 or 추가 조치가 필요한지 알려주는 세자리 숫자임
- 흔이 쓰이는 **HTTP 상태코드**들
  - `200` : 좋음. 문서가 바르게 반환됨
  - `302` : 다시보내라. 다른곳에 가서 리소스를 가져가라
  - `404` : 없음. 리소스를 찾을 수 없음
- HTTP는 각 숫자 상태 코드에 텍스트인 "사유 구절"도 함께 보냄
  - 단지 설명을 위해 포함된 것이고 실제 응답처리에는 숫자로 된 코드가 사용됨
  - ex) `200 OK` `200 Document attached` `200 Success`

### 웹페이지는 여러 객체로 이루어질 수 있음

- 애플리케이션은 보통 하나의 작업을 수행하기 위해 여러 HTTP 트랜잭션을 수행함
- 웹브라우저는 시작적으로 풍부한 웹페이지를 가져올때 대량의 HTTP트랜잭션을 수행함
- 페이지 레이아웃을 서술하는 HTML뼈대를 한번의 트랜잭션으로 가져오고, 추가로 HTTP 트랜잭션들을 수행함
- 이렇게 웹페이지는 보통 하나의 리소스가 아닌 리소스의 모음

<br>

## 메시지

- HTTP 메시지는 단순한 줄 단위의 문자열
- **요청 메시지** : 웹 클라이언트에서 웹서버로 보낸 HTTP 메시지
- **응답 메시지** : 서버에서 클라이언트로 보낸 메시지
- 세 부분으로 나뉨
- 시작줄 : 메시지의 첫줄. 요청은 무엇을 해야하는지, 응답은 무슨일이 일어났는지 나타냄
- 헤더 : 시작줄 다음의 0개 이상의 헤더필드. 각 헤더필드는 `:`으로 구분위오있는 하나의 이름과 하나의 값. 빈줄로 끝남
- 본문 : 요청의 본문은 웹서버로 데이터를 보내고, 응답의 본문은 클라이언트로 데이터를 반환함. 시작줄이나 헤더와 달리 텍스트나 임의의 이진데이터를 포함가능(이미지, 비디오, 오디오트랙 등)

<br>

## TCP 커넥션 (Transmission Control Protocol, 전송 제어 프로토콜)

### TCP/IP

- HTTP는 애플리케이션 계층 프로토콜
- HTTP는 네트워크 통신의 세부사항을 신경쓰지 않고 인터넷 전송 프로토콜인 TCP/IP에게 맡김
- TCP/IP는 어떤 종류의 컴퓨터나 네트워크든 서로 신회성있는 의사소통을 하게 해줌
- TCP는 다음을 제공
  - 오류없는 데이터 전송
  - 순서에 맞는 전달 (데이터는 보낸순서대로 도착)
  - 조각나지 않는 데이터 스트림 (언제는 어떤 크기로 보낼수있음)
- HTTP 프로토콜은 TCP 위의 계층. HTTP는 자신의 메시지 데이터를 전송하기 위해 TCP를 사용함

```
        HTTP                애플리케이션 계층
        TCP                 전송 계층
        IP                  네트워크 계층
네트워크를 위한 링크 인터페이스     데이터 링크 계층
물리적인 네트워크 하드웨어         물리 계층
```

### 접속, IP주소, 포트번호

- HTTP 클라이언트가 서버에 메시지를 전송가능하기 전에, IP 주소(Internet Protocol, 인터넷 프로토콜)와 포트번호로 클라이언트와 서버사이에 TCP/IP 커넥션이 필요
- TCP에서는 서버 컴퓨터에 대한 IP주소와 그 서버에서 실행중인 프로그램이 사용중인 포트번호가 필요
- ex)
  - `http://204.200.83.20:80/index.html` : IP주소`204.200.83.20`와 포트번호`80`을 가지고 있음
  - `http://www.naver.com:80/index.html` : IP주소 대신 도메인 이름 or 호스트명`www.naver.com`을 가지고 있음
    - 호스트명은 IP주소에 대해 이해하기 쉬운 형태의 별명
  - `http://www.naver.com/index.html` : URL에 포트번호가 없으면 기본값 `80`

#### 웹브라우저가 HTTP를 이용해서 HTML 리소스를 사용자에게 보여주는 과정

1. 웹브라우저는 서버의 URL에서 호스트명을 추출
2. 웹브라우저는 서버의 호스트명을 IP로 변환
3. 웹브라우저는 URL에서 포트번호가 있으면 추출
4. 웹브라우저는 웹서버와 TCP커넥션을 맺음
5. 웹브라우저는 서버에 HTTP 요청을 보냄
6. 서버는 웹브라우저에게 HTTP 응답을 보냄
7. 커넥션이 닫히면 웹브라우저는 문서를 보여줌
